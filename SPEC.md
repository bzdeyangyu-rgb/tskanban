# Tshuabu 节点式图像工作台 SPEC

## 1. 文档目标
本 SPEC 用于冻结当前版本（MVP）的产品范围、用户故事、交互约束、技术决策与验收标准，作为实现与测试的唯一依据。

---

## 2. 背景与目标
用户需要一个**节点式删减版**图像工作台（风格接近 ComfyUI / Coze / 麻衣画布的交互范式），重点不是“低配画布”，而是：

1. 在同一画布内完成图像生成、局部修改与多轮迭代。
2. 直接对接 image API 中转站。
3. 把提示词与每次修改过程沉淀为可检索的流程数据，为后续 RAG 做材料。

### 2.1 非目标（本期不做）
- 多人协作/权限系统。
- 云端部署与账号体系。
- 复杂插件市场。
- 通用大而全节点生态。

---

## 3. 目标用户与使用场景

### 3.1 目标用户
- 当前阶段为**单人自用**。

### 3.2 典型主流程
1. 导入一张或多张图片（图一可能是空间图，图二可能是风格参考图）。
2. 通过节点流程让系统按参考风格修改目标图。
3. 若整体结果不满意，直接重跑流程。
4. 若局部不满意，在画布中框选区域并输入局部需求，触发局部重绘。
5. 结果回到画布，继续下一轮迭代。

---

## 4. 功能需求（冻结版）

## 4.1 节点式工作流画布
- 必须提供可视节点与连线操作。
- 节点之间通过“线连接”表达数据流。
- 首版节点集合固定为 7 个（由 API 能力决定，先固定不开放自定义节点类型）。
- 连线策略：用户可自由连线（前端负责基础合法性校验，执行时再做严校验）。

## 4.2 图像输入与素材管理
- 支持三种导入方式：
  1) 文件选择
  2) 拖拽导入
  3) Ctrl+V 粘贴（满足跨平台截图后粘贴的工作习惯）
- 首版支持格式：`jpg`、`png`。
- 可对导入图像设置角色标签（如“空间图”“风格图”等，标签内容用户自定义）。

## 4.3 生成与局部重绘
- 支持 text2img 与 inpaint 两类核心执行能力。
- 局部修改方式：在画布中框选目标区域并填写局部提示词。
- 每次执行后结果图回显到画布，可继续编辑与再执行。

## 4.4 版本与迭代
- 版本策略：**保留全部版本**。
- 每次运行产生新版本，形成可追溯版本链。
- 支持基于任意历史结果继续迭代。

## 4.5 运行与失败处理
- 运行模式：整条流程一键运行。
- 失败策略：失败即停并高亮失败节点。
- 自动重试策略：
  - 仅在 API 返回报错时触发重试。
  - 每个失败节点最多重试 3 次。
  - 不基于固定耗时阈值触发重试（例如“超过两分半”不作为自动重试条件）。

## 4.6 导出
- 首版仅支持导出“当前选中的结果图”。

## 4.7 日志与RAG沉淀
本期必须记录以下信息（用户已确认保留 1/2/3/5）：
1. 流程结构（节点与连线关系）
2. 输入信息（提示词/负向词/参数/素材引用）
3. 节点输出与耗时
5. 框选坐标与局部提示词

日志用途：后续 RAG 素材，不是仅做调试。

---

## 5. 用户故事（User Stories）

1. 作为创作者，我希望导入多张参考图并标注角色，这样我可以明确“谁是空间图、谁是风格图”。
2. 作为创作者，我希望通过节点连线表达工作流，这样我可以快速调整流程而不是反复改表单。
3. 作为创作者，我希望结果不满意时一键重跑，这样我能快速得到新候选图。
4. 作为创作者，我希望只对局部区域提出修改要求并重绘，这样我不需要每次重做整图。
5. 作为创作者，我希望每轮结果都保留版本链，这样我可以回退并继续分支迭代。
6. 作为创作者，我希望失败节点自动重试三次并明确标红，这样我能少被打扰但又能快速定位问题。
7. 作为后续知识沉淀使用者，我希望系统记录结构化流程日志，这样我能直接把真实流程喂给 RAG。

## 6. 交互与体验要求

## 6.1 节点状态
每个节点需要可视状态：
- idle（未执行）
- running（执行中）
- success（成功）
- failed（失败）
- retrying（重试中，显示第几次）

## 6.2 失败反馈
- 默认少打扰。
- 必须至少包含：失败节点高亮 + 错误摘要可见。
- 自动重试过程可在节点内显示，不强制弹窗。

## 6.3 语言
- 产品界面文案与系统消息优先中文。

## 7. 技术决策（当前冻结）

## 7.1 技术栈
- 后端：Node.js + TypeScript + Express + Zod + Multer
- 前端：Web 端（当前原生实现基础上重构为节点式 UI）
- 日志：JSONL + Markdown 双写

## 7.2 对接能力
- 图像能力主通道：image API 中转站（text2img / inpaint）
- 扩展位：后续可接 GPT-5.5 作为识图工具（当前手动触发，不进自动流水线）

## 7.3 数据与持久化
- 会话与版本链：`logs/sessions/<sessionId>.json`
- 事件日志：`logs/rag_events.jsonl`
- 人读复盘：`logs/RAG_LOG.md`
- 资产输出：`outputs/...`

## 7.4 执行模型
- 前端编排节点图。
- 后端按编排结果调用现有 API。
- 节点级执行结果与耗时写入日志。

---

## 8. API 与契约（基于现有实现）
已存在并继续复用：
- `POST /api/sessions`
- `GET /api/sessions/:sessionId`
- `POST /api/upload`
- `POST /api/text2img`
- `POST /api/inpaint`
- `GET /api/logs`
- `GET /api/templates`
- `POST /api/templates`

后续为节点式执行可新增：
- `POST /api/flows/execute`（建议）
- `POST /api/flows/validate`（建议）

> 说明：若先走前端编排+现有接口串行调用，也可不立即新增 flows 接口。

---

## 9. 验收标准（MVP）

1. 用户可通过文件选择/拖拽/Ctrl+V 导入 jpg/png。
2. 画布存在节点与连线，能完成一次可执行流程。
3. 可执行 text2img 与 inpaint，结果图回显并可继续迭代。
4. 失败时节点高亮，自动重试最多 3 次，仅在 API 报错触发。
5. 会话中完整保留版本链，支持查看历史版本。
6. 可导出当前选中结果图。
7. `rag_events.jsonl` 中可看到流程结构、输入、节点输出耗时、局部框选与局部提示词。

---

## 10. 风险与约束

1. image API 响应时长与稳定性受上游影响，前端需容忍长请求。
2. 节点式 UI 改造是结构性重构，不是样式调整，前端复杂度显著高于普通画布。
3. 局部框选数据结构必须统一（坐标系、缩放换算），否则 inpaint 精度会偏移。
4. RAG 日志字段一旦上线即成为长期资产，命名要稳定，避免频繁破坏性变更。

---

## 11. 里程碑建议

### M1（节点骨架）
- 节点/连线/状态机/基础执行顺序跑通。

### M2（图像闭环）
- 导入、text2img、inpaint、局部框选与版本链跑通。

### M3（日志与导出）
- RAG字段补全、检索可用、导出可用、失败重试稳定。

---

## 12. 决策记录（简版）
- 采用“删减版节点产品”而非“低配普通画布”。
- 当前单人本机优先，先把闭环跑通。
- 重试触发条件按 API 报错，不按时间阈值。
- RAG 记录优先级高，必须随功能同步落地。
